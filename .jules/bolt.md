## 2024-06-25 - SPI Block Transfer Stack Overflows on Memory-Constrained Devices
**Learning:** Replacing byte-wise SPI loops with `SPI.transfer(buffer, size)` on hardware drivers (like SX126x/SX128x) is a massive optimization because it allows the HAL to utilize DMA or deeper hardware FIFOs. However, doing this blindly on `const` input arrays can cause crashes because the `SPI.transfer()` API is destructive (it overwrites the TX buffer with incoming MISO data). Simply allocating a local buffer array sized dynamically (e.g. `uint8_t temp[size]`) can cause stack overflows when `size` is large on memory-constrained microcontrollers.
**Action:** Always use a chunked block transfer pattern for non-destructive writes. Allocate a small, fixed-size buffer on the stack (e.g., `uint8_t temp[32]`), chunk the `size` using a `for` loop, copy chunks using `memcpy(temp, buffer + i, chunk)`, and call `SPI.transfer(temp, chunk)`. For reads, use `memset` to fill the input buffer with zeroes before block transferring it, and always guard `memcpy` and `memset` with `if (size > 0)` to prevent undefined behavior with potential null pointers.
